## 4. COMMON ERROR RESPONSES

### 4.1 Validation Error
**Status Code:** 400
```json
{
  "success": false,
  "errors": [
    {
      "msg": "Please provide a valid email",
      "param": "email",
      "location": "body"
    }
  ]
}
```

### 4.2 Unauthorized (No Token)
**Status Code:** 401
```json
{
  "success": false,
  "message": "Not authorized to access this route"
}
```

### 4.3 Forbidden (Not Approved)
**Status Code:** 403
```json
{
  "success": false,
  "message": "Your account is pending approval from admin"
}
```

### 4.4 Forbidden (Wrong Role)
**Status Code:** 403
```json
{
  "success": false,
  "message": "User role 'employee' is not authorized to access this route"
}
```

### 4.5 Not Found
**Status Code:** 404
```json
{
  "success": false,
  "message": "Resource not found"
}
```

### 4.6 Duplicate Email
**Status Code:** 400
```json
{
  "success": false,
  "message": "User already exists with this email"
}
```

## 5. TESTING WORKFLOW

### Complete Testing Sequence:

1. **Register Admin** â†’ POST /api/auth/register (role: admin)
2. **Manually Approve Admin** in MongoDB (set isApproved: true)
3. **Login Admin** â†’ POST /api/auth/login (get admin token)
4. **Register Manager** â†’ POST /api/auth/register (role: manager)
5. **Approve Manager** â†’ PUT /api/users/:id/approve (admin token)
6. **Login Manager** â†’ POST /api/auth/login (get manager token)
7. **Register Employee** â†’ POST /api/auth/register (role: employee)
8. **Approve Employee** â†’ PUT /api/users/:id/approve (admin token)
9. **Login Employee** â†’ POST /api/auth/login (get employee token)
10. **Get Employees List** â†’ GET /api/users/employees (manager token)
11. **Create Task** â†’ POST /api/tasks (manager token, assign to employee)
12. **Employee Views Tasks** â†’ GET /api/tasks/my-tasks (employee token)
13. **Employee Updates Status** â†’ PUT /api/tasks/:id/status (employee token)
14. **Add Comment** â†’ POST /api/tasks/:id/comments (any token)
15. **Get Task Stats** â†’ GET /api/tasks/stats (manager token)

## 7. TIPS FOR TESTING

1. **Save Tokens**: Keep admin, manager, and employee tokens handy
2. **Use Variables**: In Postman/Thunder Client, use environment variables for tokens
3. **Test Order**: Follow the testing sequence above
4. **Check Roles**: Test role-based access (employee shouldn't access admin routes)
5. **Validation**: Try invalid data to test validation
6. **Error Cases**: Test expired tokens, wrong credentials, etc.

## ðŸ’¡ Interview Talking Points

### Architecture Decisions

**Q: Why did you choose MERN stack?**
- Full JavaScript ecosystem (single language for frontend and backend)
- React for dynamic UI and component reusability
- Node.js for scalable, event-driven backend
- MongoDB for flexible, document-based data structure
- Large community support and extensive libraries

**Q: Explain your authentication strategy**
- Implemented JWT (JSON Web Tokens) for stateless authentication
- Token generated on login, stored in localStorage
- Token sent with every API request via Axios interceptors
- Backend middleware verifies token and extracts user info
- Admin approval adds extra security layer

### Technical Challenges Solved

**Challenge 1: Role-Based Access Control**
- Created middleware to check user roles before allowing access
- Implemented protected routes on frontend using PrivateRoute component
- Different dashboards and features for each role
- Ensured data isolation (employees only see their tasks)

**Challenge 2: State Management**
- Used Context API to avoid prop drilling
- Created AuthContext for user authentication state
- Created TaskContext for task management state
- Centralized state accessible across all components

**Challenge 3: Security Implementation**
- Password hashing with bcrypt (never store plain text)
- JWT tokens with expiration for session management
- Input validation on both client and server side
- Protected API endpoints with authentication middleware

**Challenge 4: Admin Approval Workflow**
- New users can register but cannot login immediately
- Admin must approve users from pending approvals page
- isApproved flag in database controls access
- Middleware checks approval status before granting access

### Design Patterns Used

1. **MVC Pattern (Model-View-Controller)**
   - Models: MongoDB schemas (User, Task)
   - Views: React components (UI)
   - Controllers: Express route handlers (business logic)

2. **Repository Pattern**
   - Service layer for API calls
   - Centralized data access through services
   - Easier testing and maintenance

3. **Middleware Pattern**
   - Authentication middleware (protect routes)
   - Authorization middleware (check roles)
   - Error handling middleware (catch errors)

4. **Component Composition**
   - Reusable components (Navbar, Sidebar, Modal)
   - Container/Presentation pattern
   - Props and Context for data flow

- Added proper controller function
- Exported function correctly
- Verified route with Thunder Client

### Problem 3: Token Expiration Handling
**Issue:** Users getting logged out unexpectedly  
**Solution:**
- Implemented Axios response interceptor
- Detect 401 status and clear storage
- Redirect to login page automatically
- Show appropriate error message
